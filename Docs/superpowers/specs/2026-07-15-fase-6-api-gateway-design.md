# Fase 6 — api-gateway Design

**Status:** Draft for review.

**Goal:** Build the `api-gateway` crate and turn `reactor-core` from a `/healthz`-only scaffold into TOWER's real HTTP entrypoint — every REST route the reference (`spx-portal-ref`) exposes, session-cookie auth + centralized RBAC, the OTP gate that arms auto-accept, security/CORS/rate-limit/body-limit middleware, and the first real construction of a live `PollerShared` (mounting `poller`/`ws-hub`/`notifier` as libraries into a running binary, per Fase 5's design doc's own deferral note).

**Architecture:** One new library crate, `api-gateway`, holding all route handlers/middleware/DTOs, mounted into the existing `reactor-core` binary (`axum::Router`). `api-gateway` depends on `store`, `executor`, `spx-client`, `poller`, `ws-hub`, `notifier`, `core-domain` — it is the top of the dependency graph, the one crate allowed to know about all the others. Session auth is a `tower`/axum middleware that hashes an incoming cookie token (`spx_client::crypto::session_token::hash_session_token`), looks it up in `portal_sessions` (tenant-scoped, not expired), and attaches a `CurrentUser{ tenant_id, portal_user_id, username, is_main_account }` request extension; a small `require_permission(Permission)` extractor/guard sits in front of every mutating route so the isMainAccount-style checks the reference scatters ad hoc become one central, auditable list. `store` gains real query functions for every table Fase 6 touches (today only `bookings.rs`/`quota.rs` have any) — these land in `store`, not `api-gateway`, keeping `api-gateway` free of raw SQL and matching this project's established "queries live with their table's crate" convention from Fases 1-5.

## Scope decomposition (user-approved: one design, five sequential sub-phase plans)

Each sub-phase gets its own plan doc, its own SDD task-by-task cycle, its own merge to `main` — smaller, independently reviewable units instead of one 25-30-task plan. All five share this one design doc as their source of truth.

- **6a — Foundation**: `api-gateway` crate scaffold, session-auth middleware, `CurrentUser`/`Permission`/`require_permission`, security headers + CORS allowlist + body-limit + rate-limit middleware stack, `POST /auth/portal-login`, `GET /auth/me`, `POST /auth/logout`, and — the load-bearing piece everything else needs — `reactor-core`'s `main()` actually constructing a real `PollerShared` (DB pool, `ExecutorHandle`, `SpxClient`, `SidecarClient`, `RedisPublisher`, `notifier::BotSettings`) and spawning one poller task per existing `agency_credentials` row via `poller::schedule::ensure_restored_then_spawn` (a new `store::agency_credentials::list_enabled(tenant_id)`-style query added as part of 6a itself — this bootstrap loop only needs to READ rows the Fase-2 schema already supports; it does not depend on 6b's spx-credentials CRUD *routes* existing yet, only on the table, which does), plus mounting `ws_hub::ws_router` with real session validation added (ws-hub's own upgrade handler has none today).
- **6b — spx-creds + OTP gate**: `GET/PUT/DELETE /auth/spx-credentials` (envelope-encrypted via `spx_client::crypto::envelope`), `POST /auth/spx-login` (tier 2/3 via `poller::login`), `POST /auth/request-aa-otp` + `POST /auth/verify-aa-otp` (new Redis-backed OTP generate/store/verify/rate-limit logic; WAHA delivery via `notifier::waha::send_to_waha_many` to a new `BotSettings.wa_number` field, not the group), sub-user CRUD (`/auth/portal-users*`).
- **6c — bookings + rules**: `GET /bookings/live`, `/history`, `/:id/detail`, `/spx-log`, `POST /:id/accept` (manual accept via `executor::try_claim_manual`), `PUT /bookings/settings` (rules CRUD embedded in the settings body — `sanitize_accept_rules`/`dedupe_rules`, both of which must already exist somewhere in `core-domain` per Fase 1's port of `matching.ts` — verify and reuse, do not reimplement), the OTP-gated `autoAccept:false→true` transition check, stats/health/telemetry read routes.
- **6d — prices, branding, locations, bot settings**: `GET /prices` (public) + CRUD, `GET/PUT /branding` (15MB body-limit carve-out), `GET/POST/DELETE /locations`, `GET/PUT /bot/settings` (WAHA/n8n config incl. the new `wa_number`), `GET/DELETE /bot/logs`.
- **6e — quick-accept HMAC + hardening**: build the Rust `sign_quick_token`/`verify_quick_token` using the already-reserved `LABEL_QUICK_ACCEPT_HMAC` subkey (`spx_client::crypto`), `GET /q/:token` + `POST /q/accept` (long-lived HMAC token, no login), the short-code Redis flow (`GET/POST /accept/:code`), both mounted OUTSIDE the session-auth middleware per the reference; final Fase-6 workspace verification + DoD sign-off.

## Global constraints (binding across all five sub-phases)

- **Runtime/binary assembly**: the master spec's "two separate Tokio runtimes" (HOT pinned-core vs CONTROL) is an **explicit, disclosed deferral to Fase 8**. Fase 6 spawns everything on the default multi-thread runtime `reactor-core` already has — Aturan Keras #1 ("parity first, optimization second") and this project's consistent Fase-1-through-5 preference for correctness-then-performance apply directly: a literal `core_affinity`-pinned second runtime is a hardening/observability concern (Fase 8 already owns "Bench criterion decision-path di CI", soak testing, SLO dashboards) not a Fase-6 blocker. Fase 6 must NOT attempt runtime-splitting; if a reviewer flags this as a plan gap, the answer is "intentionally out of scope, see Fase 8."
- **RBAC model**: the reference has no granular permission table — every gate reduces to `portal_users.is_main_account`. `Permission` is therefore a small enum (`ManageSubUsers`, `ManageSpxCredentials`, `ManageBotSettings`, `ArmAutoAccept`, `ManagePrices`, `ManageBranding`, …) whose `require_permission` implementation today is uniformly "must be main account" — the payoff of centralizing it now (vs. the reference's scattered `if (!session.isMainAccount)` checks) is that a future finer-grained ACL only touches one function, not N call sites. Do not build a database-backed permission table — that is not in the reference and would be scope creep.
- **Session auth**: opaque bearer/cookie token, `spx_client::crypto::session_token::generate_session_token()`/`hash_session_token()` (Fase 3, already shipped) — store only the hash in `portal_sessions.token_hash`, exactly as that module's doc comments already describe. Every tenant-scoped query MUST run inside `store::begin_tenant_tx` (sets `app.tenant_id` for RLS) — a query issued outside that silently returns zero rows against `app_role`, and the master spec's Fase-2 note flags this exact class of silent-no-op bug as the reason Fase 6 is where a non-superuser DB role must actually be used in the connection pool (`app_role`, not `tower`). Fase 6 Task 6a must configure `reactor-core`'s production pool to authenticate as a non-superuser role.
- **Rate limiting / CORS / security headers**: master spec names `tower-governor`; none of `tower-governor`/`tower_http::cors`/security-header middleware exists anywhere in this workspace yet (confirmed via research — zero hits). This is a clean-slate integration in 6a: `tower_http::cors::CorsLayer` with an explicit origin allowlist (`Vec<HeaderValue>`, no wildcard — the reference's own code comments flag an earlier ngrok-wildcard CSRF near-miss, learn from it), a small custom tower `Layer` for the 6 fixed security headers PLUS a real `Content-Security-Policy` header (the reference has NO CSP header at all — master spec explicitly requires "+ CSP", so this is a genuine improvement over the reference, not parity), `tower_http::limit::RequestBodyLimitLayer` (1.5MB default, 15MB carve-out scoped to exactly the branding `PUT` route via a per-route layer override, not global), and `tower_governor` for the public-GET rate limit (120/min/IP) plus a stricter per-route limit on the quick-accept endpoints (12/min accept, 60/min view, matching the reference's own numbers — no reason to diverge).
  - **Forward note for Fase 8's VPS overlay (Traefik) — review finding, tracked here since Traefik doesn't exist yet:** Task 8's `POST /auth/portal-login` rate limiter uses `tower_governor::SmartIpKeyExtractor`, which trusts the leftmost `X-Forwarded-For` value with no validation that it came through a trusted hop (see `Backend/crates/api-gateway/src/middleware/rate_limit.rs`'s doc comment). This is only safe because `Docker/Caddyfile` is configured to OVERWRITE `X-Forwarded-For` with the real peer address (`header_up X-Forwarded-For {remote_host}`) rather than append to a client-supplied value. **Fase 8's Traefik config for the VPS overlay MUST replicate this same overwrite behavior** — configure `forwardedHeaders.trustedIPs` correctly (scoped to the actual upstream/edge hop), NOT `forwardedHeaders.insecure` (which passes client-supplied forwarded headers through unchecked and would reopen this exact gap). Verify this invariant as part of Fase 8's DoD before the login rate limiter is trusted in production.
- **Testing convention (unchanged from Fases 1-5)**: real Postgres (`127.0.0.1:15432`) + real Redis (`127.0.0.1:16379`) for anything touching DB/dedup state, `wiremock` for outbound SPX/WAHA/sidecar HTTP, `tokio::time::pause`/`advance` for pure timing logic (OTP TTL/rate-limit windows, quick-token expiry), real HTTP requests against a locally-bound `axum::serve` test server for route-level integration tests (mirroring `ws-hub`'s Task 12 `local_broadcast.rs` pattern) rather than calling handler functions directly, so middleware ordering (auth → RBAC → rate-limit → handler) is genuinely exercised.
- **Secrets discipline (unchanged from Fase 3)**: session tokens, SPX credentials, WAHA keys stay `SecretString`/envelope-encrypted end-to-end; passwords keep using `spx_client::crypto::password::verify_password`'s constant-time comparator, exactly as before. OTP codes are short-lived Redis values (not secrets requiring `SecretString`, but must never be logged). **Accepted deviation, reviewed at Task 4 and again at the Fase 6b whole-branch review:** `api-gateway::otp::verify` compares the stored vs. submitted 6-digit OTP code with a plain `==`/`!=`, NOT the reference's `timingSafeEqual` constant-time compare. This is deliberate, not an oversight: a 6-digit numeric code has vastly lower entropy than a password (1,000,000 possibilities vs. argon2id's much-higher-entropy, much-longer-lived secret), and the 5-attempt cap — enforced BEFORE the comparison ever runs — is this code's dominant defense; even a perfectly-informative timing side-channel only saves an attacker attempts they don't have (5 total, ever, per code). Upgrading to a constant-time comparator here would be over-engineering against an already-dominant defense. This carve-out applies ONLY to the OTP code comparison — passwords and every other secret comparison in this codebase still require a constant-time comparator.
- **`is_coc`/matching-engine reuse**: rules CRUD (6c) must reuse Fase 1's `core-domain` rule engine and Fase 2's generated-column schema as-is — Fase 6 is a transport layer over existing business logic, it does not re-derive or duplicate matching/dedup rules.
- **Tenant resolution (self-review addendum): single configured deployment tenant, not multi-tenant-by-request routing.** The reference (`spx-portal-ref`) is genuinely single-tenant (Fase 2's own design note: "repo acuan single-tenant... skema ini desain baru, bukan porting"). `tenants.slug` exists as forward-looking RLS infrastructure, but nothing in the master spec or the reference requires resolving "which tenant" from request input (no subdomain routing, no tenant selector on the login form). Fase 6 resolves the ONE deployment tenant from a `TENANT_SLUG` (or `TENANT_ID`) env var at `reactor-core` boot, looks up its row once, and holds the resolved `tenant_id` in `AppState` for every handler to use — login, session lookup, and every other route all operate against this single fixed tenant. Building per-request tenant resolution (by subdomain, header, or login-form field) is explicitly OUT of scope — it's infrastructure with no current consumer and no reference behavior to port; add it later if TOWER genuinely needs multiple deployment tenants sharing one `reactor-core` process, which is not today's requirement (today's requirement is one deployment per freight company, matching the reference's own single-tenant model).
- **Data-visibility model within one tenant (6a Task 10 review finding, tracked here): every `portal_user` under a tenant can see every `agency_credentials` account under that SAME tenant — there is no per-user account-scoping, and this is intentional, not an oversight.** `ws-hub`'s per-account WS channel (`acct:<account_id>`, Fase 5) has no tenant dimension in its naming at all — it was designed for a single-tenant deployment where this doesn't matter. Task 10 added real session VALIDATION to the WS upgrade (a request needs a genuine, unexpired session — closing the Fase-5 "any `?session=`/`?account=` string works" gap), but did not add per-account AUTHORIZATION (checking that the validated session's user is entitled to the specific `?account=` requested) — deliberately, since authentication and authorization are different concerns and the brief scoped Task 10 to authentication only. Given today's single-tenant-per-deployment model (see above), the practical exposure is: any logged-in staff member of ONE company (tenant) can view any OTHER account belonging to that SAME company via the WS feed — which mirrors the reference's own actual sub-user permission model (sub-user restrictions are about which SETTINGS a user may change, e.g. `is_main_account`-gated mutations, never about which of the company's OWN SPX accounts' live data a logged-in staff member may view). Cross-TENANT leakage is not possible today (there is only one tenant per `reactor-core` process). If a future requirement genuinely needs per-user account-scoping WITHIN one tenant (not in the reference, not in today's requirements), that needs a new `portal_user`-to-`agency_credentials` authorization mapping — out of scope for Fase 6, tracked here so it isn't mistaken for an unexamined bug.

- **WS auth channel is currently unreachable by a browser client (Fase-6a whole-branch review finding, tracked here — must resolve before Fase 7 depends on it).** Task 10's `ws_handler_with_auth` validates ONLY a `?session=` query parameter; `portal_login` (Task 5) deliberately sets the session token as an `HttpOnly` cookie and never returns it in a JSON body (a correct security choice on its own). The consequence: a real browser frontend has no way to read the plaintext token to construct `ws://…/ws?session=<token>` — the auth channel used by the WS upgrade and the auth channel a browser can actually supply (the cookie, sent automatically) are two different things. The reference validates the cookie directly on its own upgrade handler; TOWER has not yet matched that. This does not block 6a's own DoD (no frontend exists yet to hit the gap) but MUST be resolved before Fase 7's Command Center UI needs a live WS connection. Two candidate resolutions, to be decided when Fase 7 (or a 6b/6e follow-up) actually needs this: (a) have the validated WS upgrade ALSO accept the token from the `HttpOnly` cookie (mirrors the reference, reuses the existing session directly, but puts the same long-lived 12h bearer token in a URL-adjacent request that CAN'T be logged since it's a cookie not a query string — probably the simpler, safer choice); (b) mint a short-lived, single-use WS-ticket (a natural fit alongside 6b's Redis-backed OTP work) that IS safe to expose to JS and put in a query string, avoiding a long-lived token ever appearing in Caddy access logs/browser history/referrers. Do not let 6b/6e each assume the other owns this decision — it needs to be made explicitly.
- **Manual accept (`POST /bookings/:id/accept`, 6c) is session-gated only — no `require_permission` — while the auto-accept arm transition (`PUT /bookings/settings`, 6c) requires `ManageRules`+`ArmAutoAccept`+a single-use OTP proof (Fase-6c whole-branch review finding, tracked here — not a merge blocker, but wants an explicit product confirmation before Fase 7's UI leans on it).** This is a deliberate, disclosed design decision (Task 10's own plan-doc scope note, reviewed and accepted at both the task-level review and this whole-branch review): claiming an individual, already-visible booking is treated as "operating the queue" (any logged-in staff member of the tenant may do it, matching this doc's own data-visibility model above), while flipping the GLOBAL `auto_accept_enabled` kill switch is treated as "administering settings" (main-account-gated, OTP-proofed). The asymmetry is coherent with that stated model and is a one-line reversal if wrong (`require_permission(&user, Permission::ManageRules)` as `accept`'s first statement) — but it only became fully visible once every 6c route existed side-by-side, so it's tracked here for a conscious go/no-go before Fase 7 builds a UI that assumes any staff member can one-click-accept a ticket. If the product decision is "no, gate it," make the one-line change; if the decision is "yes, this is intended," this note can be removed once acknowledged.
- **`docker compose up` on a clean checkout cannot yet bring `reactor-core` to a healthy state (Fase-6a whole-branch review finding, tracked here).** Task 9 switched `reactor-core`'s production DB connection to `app_role` (non-superuser, per correction #2 below) and requires two operator steps that exist only as documentation today, not as committed automation: (a) migrations must run as the superuser `tower` (this project has never had `reactor-core` itself run `store::run_migrations` — that has always been a separate dev/deploy-tooling concern) — `app_role` has no DDL privilege and `build_state()` never calls it; (b) `app_role`'s password must be set out-of-band via `ALTER ROLE app_role PASSWORD '<from APP_ROLE_PASSWORD>'`, since migration `0019` deliberately grants only `LOGIN` capability, never a committed password (Aturan Keras #5). Design doc DoD #4 ("`docker compose up` end-to-end smoke test passes") is Fase-6-WIDE, not 6a's sole responsibility, and 6a's own tests already exercise the full `app_role` path successfully (via a test helper that sets the password and runs migrations) — so this is a genuine, disclosed deployment-automation gap, not a code defect. Resolution (a committed one-shot init step — an init container, or a documented deploy script wired into `docker-compose.yml` — that runs migrations as `tower` then sets `app_role`'s password from the `APP_ROLE_PASSWORD` secret) was explicitly confirmed as **Fase 8's** responsibility (human decision, Fase 6e planning session) — NOT part of 6e's "hardening" scope, which turned out to mean "quick-accept HMAC + the final Fase-6 DoD sign-off," not general deployment automation. Fase 6e's own DoD cross-check does not claim this item closed; do not assume `docker compose up` is production-ready until Fase 8 closes it.
- **Corrupt `site_settings` row handling diverges between routes (Fase-6d whole-branch review finding, tracked here — disclosed, non-blocking).** `GET /branding` (`routes/branding.rs`) does `serde_json::from_value(v).unwrap_or_default()` — a malformed `price_page` row silently degrades to `Branding::default()`. `GET/PUT /bot/settings` (`routes/bot.rs`) maps the same class of corruption in its own `site_settings` row to `ApiError::Internal` (500). Both are individually defensible (a public page degrading gracefully vs. an admin page surfacing a real problem) but the divergence between two tasks handling the identical failure mode was not a deliberate joint decision — track it as a disclosed, non-blocking observation for whoever next touches either route.
- **Public routes (`GET /prices`, `GET /branding`) read `AppState.tenant_id`, not a session-derived tenant (Fase-6d whole-branch review finding, tracked here).** There is no session on a public route to derive one from, so these routes only ever serve the single tenant configured at that `AppState`/deployment instance. This is coherent with TOWER's existing single-tenant-per-instance deployment model (see `.env.example`'s `TENANT_SLUG`, and this doc's own "Tenant resolution" note above) but is a conscious constraint worth naming explicitly before any future multi-tenant-single-gateway deployment is considered. Related, same review: auto-accept wins record a `bot_log` audit entry (Fase 6d Task 7) but Fase 6c's manual-accept path does not — an audit-trail asymmetry, correctly out of scope for a fix in this sub-phase, noted for whichever future phase revisits audit completeness.
- **Quick-accept short-code *generation* is unbuilt (Fase 6e planning-research note, tracked here — disclosed, non-blocking).** `notifier::notify_new_tickets` (built Fase 5) has no caller anywhere in the codebase, and its link-building still formats a raw `spx_id` into the WA message rather than a real `spx:qa:<code>` short code — Fase 6e built the READ side of both the HMAC-token (`GET/POST /q/:token`) and short-code (`GET/POST /accept/:code`) quick-accept flows (genuine parity with the reference, which itself also has `signQuickToken`/`verifyQuickToken` as effectively-dead-until-linked infrastructure — confirmed via research, `signQuickToken` has no call site anywhere in the reference repo either), but generating and sending a real link from a real new-ticket-detection event is unbuilt. Both `GET/POST /q/:token` and `/accept/:code` are reachable, correct, rate-limited, and tested end-to-end (including a real browser-functional confirmation page — CSP-compatible, HTML-escaped) — but nothing in TOWER today produces a link to them. Tracked for whichever future phase wires `poller`'s ticket-detection loop to `notifier::notify_new_tickets` with a real short-code generator (mirrors the reference's `webhook.ts::genAcceptCode`) and/or a `sign_quick_token` call site.

## Corrections vs. the reference (intentional deviations, disclosed up front — mirrors Fase 5's numbered-corrections pattern)

1. **Real CSP header** where the reference has none (master spec explicitly requires it) — an improvement, not parity-breaking, since CSP only *restricts* the browser's behavior and the reference never relied on its absence for anything functional.
2. **`app_role` (non-superuser) in the production connection pool**, not `tower` — closes the Fase-2-flagged silent RLS-bypass gap; the reference has no RLS at all to compare against (single-tenant), so this is a TOWER-specific hardening with no reference behavior to diverge from.
3. **Two-Tokio-runtime split deferred to Fase 8** (see Global Constraints) — the reference is single-process Bun/Node with no equivalent runtime-splitting concept at all; TOWER's own master spec places the *literal pinning* work under Fase 8's hardening scope textually ("Bench criterion... fail regres >20%"), so Fase 6 building working, correct routes on the default runtime is the correct build order (parity/correctness before this specific optimization), not a deviation from the reference (which has no runtime model to be faithful to).
4. **Centralized `require_permission` enum** replacing the reference's scattered inline `isMainAccount` checks — behaviorally identical (still boolean, still main-account-gated) but structurally different, per the master spec's own explicit instruction ("RBAC require_permission terpusat").
5. **`store` gains real query functions per table** (today almost entirely missing) rather than `api-gateway` writing raw SQL inline — an internal Rust-workspace organization choice with no reference equivalent (the reference has no crate-boundary concept).

## Testing

Every sub-phase's route-level tests spin up a real `axum::serve` instance (random port), drive it with a real HTTP client (`reqwest`/`wreq`), and assert on full response status + body + relevant headers (e.g. CORS/CSP presence, rate-limit 429 after N requests) — never call handler functions directly, since middleware correctness (ordering, header propagation) is exactly what would go untested that way. DB-touching routes seed real Postgres with a unique tenant id per test (established convention). OTP/rate-limit windows use `tokio::time::pause`/`advance` wrapped around the *logic* (extracted into a testable pure function, mirroring `poller::notif_watch`'s `next_backoff` precedent) rather than real sleeps, except where a real-socket accommodation is genuinely needed (same documented exception class as Fases 5's `local_broadcast.rs`/watchdog tests).

## Definition of Done — Fase 6

1. All five sub-phases (6a-6e) merged; every reference route in the research brief has a TOWER equivalent (route-level test proving status + body shape).
2. Session auth + `require_permission` enforced on every mutating route; a route-level test proves an unauthenticated request is rejected and a non-main-account request is rejected on every main-account-gated route.
3. OTP gate: `autoAccept:false→true` genuinely blocked without a fresh, single-use, main-account-scoped OTP proof; a stale/reused/wrong-account proof is rejected (real Redis test).
4. `reactor-core` boots, connects to Postgres as a non-superuser role, constructs a real `PollerShared`, spawns account tasks, serves `/healthz`, the REST API, and `/ws` from one binary; `docker compose up` end-to-end smoke test passes.
5. Security headers (incl. real CSP) + CORS allowlist + body-limit (1.5MB default / 15MB branding carve-out) + rate-limit all present and tested.
6. Quick-accept HMAC token: `sign_quick_token`/`verify_quick_token` round-trip correctly, a tampered/expired token is rejected (constant-time compare verified, not just "works on the happy path").
7. `cargo test`/`clippy`/`deny` clean workspace-wide; no new license/advisory exceptions beyond what's already justified in `deny.toml` unless a genuinely new one is needed and disclosed with the same rigor as Fase 5's `RUSTSEC-2023-0071`.
8. `cargo tree -p api-gateway` confirms it is the only crate depending on ALL of `store`/`executor`/`spx-client`/`poller`/`ws-hub`/`notifier`/`core-domain` — no other crate gains a new cross-dependency as a side effect of this fase.
